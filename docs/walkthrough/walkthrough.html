1 · What this platform is

ThinkAiSolutions is a multi-tenant WhatsApp Business API reseller platform. WhatsApp messaging isn't sold directly by Meta to most businesses — it flows through a chain of providers. This platform's job is to let ThinkAiSolutions become a reseller in that chain: buy WhatsApp capacity wholesale and resell it to its own clients (clinics, pharmacies, labs, hotels, restaurants) under its own brand, dashboard, billing, and markup.

The supply chain — today vs. the goal

The platform inserts ThinkAiSolutions as the branded reseller layer between the BSP and the end clients.

How ThinkAiSolutions makes money

  • Meta charges per message, by category: marketing utility authentication are paid; service replies inside a 24-hour window are free.
  • ThinkAiSolutions pays the BSP a wholesale cost rate, and sets a higher charge rate for each client. The difference is the margin.
  • Clients pre-pay into a wallet. Every billable message is debited from that wallet automatically. When the wallet is empty, sending is blocked.
  • All money is stored as integer paise (never floating-point rupees) to avoid rounding errors.
Multi-tenant from day one. Each client is a "tenant". Every piece of data — contacts, messages, wallet, pricing — is scoped to a tenantId, and one tenant can never see another's data.

2 · The big picture — system architecture

The platform is a small number of cleanly separated pieces. The browser only ever talks to our own API; it never touches the BSP or payment secrets directly.

4 · Logging in

Everyone starts at the same login screen. There is no public sign-up — accounts are created for clients by the reseller admin (see §5).

What happens when you click "Sign in"

  1. Firebase Auth checks the password. The browser sends email + password to Firebase Authentication and gets back a signed ID token.
  2. Role & tenant are read from the token. The token carries custom claims (role, tenantId) that the server set when the account was created. No extra database lookup is needed.
  3. You're routed to your home. Reseller admins go to /admin; tenant admins and agents go to /dashboard.
  4. Every later API call carries the token. The app attaches Authorization: Bearer <token> to each request; the server verifies it and enforces the role/tenant on every endpoint.
Why this is safe: the token is cryptographically signed by Firebase and verified on the server with the Firebase Admin SDK on every request, including a check that the token hasn't been revoked.

5 · Reseller admin — onboarding a client

This is the most important admin journey: turning a new client into a live tenant who can send WhatsApp messages. It's a 4-step wizard at /admin/tenants/new. Crucially, the tenant is created at step 1 and the same tenantId is carried through steps 2–4 — so the admin never types an ID by hand.

7 · The wallet — recharge & billing

Each tenant has one prepaid wallet (tenant_admin only — agents can't see it). The balance funds every billable message.

Illustration The balance card and the recharge modal's GST preview.

The GST rule (important): 18% GST is charged once, at recharge, on top of the credit. So if a client wants ₹1,000 of sending power, they pay ₹1,180 and the wallet is credited ₹1,000. Per-message debits afterwards are the bare rate with no further GST — this avoids charging GST twice.

How recharge actually works

  1. Client enters an amount (or taps a quick button: 500 / 1000 / 2500 / 5000). The modal previews credit + GST + total.
  2. Server creates a Razorpay order for the total (credit + GST) and saves a walletOrder record with the exact amounts. POST /api/wallet/recharge/order
  3. Razorpay Checkout opens in the browser; the client pays by card / UPI / netbanking.
  4. The browser does NOT credit the wallet. It just shows "Payment received — balance will update shortly" and starts polling.
  5. Razorpay calls our webhook (payment.captured). The server verifies the signature, looks up the saved order (amounts come from our record, never the webhook), credits the wallet, writes a transaction, and generates a GST invoice — all idempotently (a repeated webhook can't double-credit).
  6. The balance updates and the transaction history shows the recharge.
Why credit only on the verified webhook? A browser can be tampered with; the signed server-to-server webhook is the only trustworthy confirmation that money actually arrived.

8 · How a message is sent & billed

This is the heart of the platform. When a tenant sends a template, the order of operations is deliberate: debit first, send second, refund if the send fails — so a client is never charged for a message that didn't go out, and can't send for free.

REFUND the debit (client pays nothing).

The rules baked in

  • Debit is atomic. Reading the balance, checking it, decrementing it, and writing the ledger row all happen in one Firestore transaction — no double-spend, even under concurrent sends.
  • Insufficient balance blocks the send with a clear 402 insufficient_funds; nothing is sent.
  • Idempotent by messageId. Each debit is keyed to its message, so a retried request can't charge twice.
  • Free-text replies are different: allowed only while the 24-hour window is open, and they cost nothing (no debit).
  • Delivery status (sent → delivered → read, or failed) arrives later via the BSP webhook and updates the message — only ever moving forward, never backward.

9 · How a reply comes back in

When a customer replies on WhatsApp, the BSP sends it to our webhook. This both stores the message and opens (or extends) the 24-hour service window.

The 24-hour service window — why it matters

  • The window opens only on an inbound message from the customer, and lasts 24 hours.
  • While open: agents can reply with free text, for free.
  • While closed: the only way to reach the customer is an approved template, which is billed.
  • This mirrors Meta's own rules and is why templates (and the wallet) matter for re-engaging customers.
Idempotency everywhere: webhooks can be delivered more than once. Every write is keyed (by message id, payment id, or event id) so duplicates are harmless — no double messages, no double charges.

10 · Data model — what is stored where

Everything lives in Firestore. Almost all data is nested under tenants/{tenantId}/… so isolation is structural. The browser never reads Firestore directly — it goes through the API — but the security rules still lock everything down as defence in depth.

CollectionHolds
tenants/{id}Client profile: name, plan, status, billing (GSTIN, state, address).
users/{uid}Each login's role, tenantId, name, email.
tenants/{id}/wabas/{wabaId}Connected WhatsApp numbers + references to the encrypted apikey / webhook secret (never the secret itself).
tenants/{id}/contactsCustomers: phone, name, tags, opt-in status.
tenants/{id}/templatesApproved/pending templates synced from the BSP.
tenants/{id}/conversationsOne thread per customer + the windowExpiresAt timestamp.
tenants/{id}/messagesEvery message in/out: direction, status, category, costPaise, bspMessageId.
tenants/{id}/wallet/currentThe single balance doc (balancePaise).
tenants/{id}/walletTransactionsThe ledger: recharge / debit / refund, with balance-after.
tenants/{id}/walletOrdersRazorpay orders awaiting payment (server-only).
pricing/{id} & pricingCost/{id}Charge rates (client-visible) and your cost rates (admin-only) per tenant.
invoices/{paymentId}GST invoices with the CGST/SGST or IGST split.
secrets/{ref}AES-256-GCM encrypted apikeys & webhook secrets. Deny-all to clients.
processedEvents/{key}Webhook dedup markers (idempotency).
Authorization is O(1): the rules read role and tenantId straight off the signed token — no extra database lookups — so a tenant member can only ever read their own subtree, and reseller-only data (cost rates, secrets, orders) is never exposed to tenants.

The security model in one view

  • Secrets stay server-side. BSP apikeys, the Razorpay secret, the encryption key, and Firebase admin credentials live only on the server — never in the browser bundle, never in a Firestore document in plaintext.
  • The browser bundle is public by design. Only safe values ship to it (the Firebase web config and the Razorpay public key id), and access is gated by Auth + rules.
  • Webhooks are verified before trust. Razorpay by HMAC-SHA256 signature; the BSP by a constant-time secret-header compare. Unsigned/forged calls are rejected.
  • Money is paise, and atomic. Integer paise throughout; all balance changes run in Firestore transactions; all credits/debits are idempotent so retries are safe.
  • Tenant isolation is enforced server-side. A tenant's scope comes from their token, never from the request body — they can't smuggle another tenant's id.
  • Apikeys are encrypted at rest (AES-256-GCM) and only decrypted in memory at the moment of a send.

Current status & what's next

The app is live — client on Firebase Hosting, server on Cloud Run (asia-south1). Login, tenant onboarding, WABA connect, template send, inbox, the 24-hour window, and wallet/Razorpay billing are all built (Phase 1 / MVP).

Deliberately out of scope for now (later phases): a template builder & approval flow, broadcast campaigns, a no-code chatbot, contact import/segments, Meta embedded signup, vertical packs (Clinics / Restaurants), AI chat/voice, and white-label branding.

12 · Glossary

TermMeaning
BSPBusiness Solution Provider — a company (like Pinnacle) authorised to sell access to Meta's WhatsApp API.
ResellerA layer that buys from a BSP and resells to end clients. ThinkAiSolutions aims to be this layer (replacing Twincles).
TenantOne client business on the platform. All their data is isolated under one tenantId.
WABAWhatsApp Business Account — the Meta object a phone number lives under.
TemplateA pre-approved message format required to start (or re-open) a conversation. Billable.
Service windowThe 24-hour period after a customer messages you, during which free-text replies are free.
Paise1/100 of a rupee. All money is stored as whole paise to avoid float errors.
GSTIndian 18% tax, charged once at wallet recharge; split into CGST+SGST (same state) or IGST (different state).
IdempotentAn operation that's safe to repeat — running it twice has the same effect as once (used for webhooks & billing).
Custom claimsRole & tenant data baked into the login token so the app knows who you are without extra lookups.
That's the whole system. A client logs in → (admin onboarded them with a WABA, pricing, and a login) → they top up a wallet → send approved templates that are billed per message → customers reply within a 24-hour window where free-text is free → everything is stored per-tenant, with secrets encrypted and every webhook verified.
